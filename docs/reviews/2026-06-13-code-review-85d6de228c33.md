## Findings

### High — Cached random solutions become stale after keyboard movement

File: `index.html`

- `state.cachedSolution` is reused directly when `РЕШЕНИЕ` is clicked: `index.html:1117`.
- Mouse/button position edits clear the cache: `index.html:1078`.
- Dependency edits clear the cache: `index.html:1098`.
- Keyboard movement in config does not clear the cache: `index.html:1414` and `index.html:1422`.

Impact: after generating a random puzzle, the user can press `A` or `D` in Config to change the active plate position, then click `РЕШЕНИЕ`. The app will reuse the old cached BFS solution for the previous positions and enter Solve with an invalid solution path.

Recommendation: clear `state.cachedSolution = null` whenever a config-stage keyboard move succeeds, directly beside `state.history.push(...)`.

### Medium — Import validation accepts malformed dependency graphs that can poison solver/render state

File: `index.html`

- Import validation only checks plate-level primitive fields and that `deps` is an array: `index.html:1155`.
- It does not validate unique contiguous plate ids, dependency object shape, valid `targetId`, valid `direction`, numeric positive `steps`, duplicate ids, or uniform `positions`.
- Movement and BFS trust dependency fields: `index.html:665`, `index.html:1507`.

Impact: pasted JSON can introduce `undefined`, string, `NaN`, out-of-range, or huge target ids. Examples include `steps: "x"` or `targetId: 999`. That can produce `NaN` positions, sparse arrays, invalid CSS transforms, misleading matrix state, or very slow solver behavior.

Recommendation: normalize imported configs before applying:

- require ids to be exactly `1..N`;
- require all plates to share the same odd `positions`;
- require every dep to have existing non-self `targetId`;
- require `direction` in `same|opposite`;
- require integer `steps >= 1`;
- reject or deduplicate repeated target deps.

### Medium — BFS queue and path copying will scale poorly on larger hard/random searches

File: `index.html`

- BFS consumes the queue with `queue.shift()`: `index.html:1538`.
- Each branch copies the full path with `[...path, move]`: `index.html:1561`.

Impact: `shift()` is O(n) on JS arrays, and copying the full path for each enqueued state multiplies memory pressure. The spec allows up to roughly `7^8` states, so hard random generation can become much slower than the theoretical BFS bound and may spend most time in array churn.

Recommendation: use a head index for the queue and store parent pointers instead of copying full paths:

- `let head = 0; while (head < queue.length) { const node = queue[head++]; }`
- Store `{ positions, parentIndex, move }`.
- Reconstruct the path only after finding the goal.

### Medium — Main-thread and worker BFS APIs have drifted

File: `index.html`

- Main-thread `bfsSolve()` returns `String[] | null`: `index.html:731`.
- Worker `bfsSolve()` currently returns `{ solution, iters }`: `index.html:1526`.

Impact: runtime solve/random uses the worker path today, so this is not immediately user-visible. It is still a maintenance trap: the same function name now means different return shapes in the same file. A future fallback, test, or refactor can easily call the wrong version and break.

Recommendation: either make both return the same shape or rename them explicitly, e.g. `bfsSolve()` and `bfsSolveWithStats()`. Add a tiny syntax/contract smoke test for both.

### Low — Worker Blob URLs are never revoked

File: `index.html`

- `createWorker()` creates a Blob URL inline: `index.html:1646`.
- On cancel, the worker is terminated and recreated: `index.html:1710`.
- The Blob URL is not stored or revoked.

Impact: repeated cancel/retry cycles leak Blob URLs for the session. The leak is small, but it is easy to avoid.

Recommendation: store the URL returned by `URL.createObjectURL(...)` and call `URL.revokeObjectURL(url)` after worker creation or when replacing the worker.

## Open Questions

- Should random difficulty be based on compressed solution length or raw move count? The current code filters on compressed notation length, so `1D7` counts as one step for difficulty.
- Should import support non-`1` dependency `steps`, or should the schema reject anything except `steps: 1` until the matrix UI can edit larger step values?
- Should the main-thread BFS helper remain at all if the worker is the only supported execution path?

## Test Gaps

- No automated tests cover stale cache invalidation across all config mutation paths.
- No tests validate import rejection for malformed dependency objects.
- No performance regression guard exists for hard random generation or large BFS searches.
- No browser smoke test covers worker cancel/recreate behavior.

## Summary

The app is coherent for a single-file prototype, and the worker split is the right direction for keeping the UI responsive. The highest-risk correctness issue is stale cached random solutions after keyboard movement. The next most important hardening area is import validation, because malformed JSON can break assumptions across movement, rendering, and BFS.
